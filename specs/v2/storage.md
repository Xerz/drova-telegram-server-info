# Storage Spec

## Database

Use SQLite as the default local database.

Default path:

```text
data/drova_bot.sqlite3
```

Schema is managed by migrations.

## Tables

| Column | Type | Notes |
| --- | --- | --- |
| `telegram_chat_id` | integer primary key | Telegram chat id. |
| `drova_user_id` | text nullable | Account UUID. |
| `encrypted_proxy_token` | blob nullable | Encrypted token. |
| `selected_station_id` | text nullable | Null means all stations. |
| `session_limit` | integer not null default 5 | Valid range `1..100`. |
| `timezone` | text not null default `Asia/Yekaterinburg` | IANA timezone. |

| `updated_at` | datetime | UTC. |

## Token Security

- Tokens are encrypted at rest.
- Required env: `BOT_SECRET_KEY`.
- Key format: implementation may use Fernet-compatible base64 key.
- If key is missing in production mode, bot must fail startup.
- Logs and errors never include token material.

## Migration From Legacy

V2 may provide a one-time import command:

```text
python -m drova_bot.tools.import_legacy persistentData.json
```

Rules:

- Import `authTokens`, `userIDs`, `limits`, `selectedStations`, `stationNames`.
- Encrypt imported tokens before writing SQLite.
- Validate `limits`; invalid values become default `5`.
- Preserve all-stations selection as `selected_station_id = null`.
- Never delete legacy file automatically.

## Cache Policy

- Product cache refreshes after successful token connect and when unknown product ids appear.
- Station cache refreshes after `/token`, `/station`, `/current`, publish changes.
- Cache failures are logged and surfaced only when they block the requested command.
